Developer Notes – Personal Background

A personal note about how OTAB started, how the project evolved, and how AI-assisted tools were used during development.

As a teenager I started programming with the Warcraft III map editor. Later I built my first personal websites and small tools.

For the game World of Warcraft I created small addons to customize the UI the way I wanted to play.

At some point I even wrote my own comic book reader for .cbr and .cbz files because other readers were simply too heavy for my old and rather terrible tablet.

I also once built a small 3D rendering engine in PHP that was able to render one whole frame — mostly as an excuse to learn algebra that I never had the chance to properly study in school.

For the game Grim Dawn I created a small tool to manage multiple in-game stashes. The code design was honestly quite ugly — it relied on two third-party injection tools because I had no idea how to hook into DirectX rendering or how to write proper C++ code.

I am not a trained software developer. Programming is simply something I enjoy doing. I might struggle to visualize something simple like a tree with an apple in my head — but a multidimensional array passing through a generic method somehow makes perfect sense to me.

Over time I picked up knowledge in many different languages: PHP, JavaScript, Lua, SQL, Batch, BASIC, C#, and Java (for Android). All of it purely out of curiosity and hobby interest.

And now, through OTAB, Unity has joined that list as well.

For many years Stack Overflow was my primary place to learn things when I got stuck.

Today, tools like AI exist.

Why I started OTAB

The mod AllTameable Taming Overhaul is a fantastic mod and deserves a lot of credit. However, for my own experiments it did not provide the level of configurability I was looking for.

Creatures in Valheim are very different from each other. They live in different environments, have different AI behaviors, appearances, factions, groups, and even their own lore background. Because of this I wanted a system that allowed creatures to be configured in a much more detailed way.

OTAB originally started as a very small personal mod. At that time I had very little Unity knowledge and did not use AI at all. Much of the early code relied heavily on reflection, without delegates, and used a large number of WeakReference objects.

Later I began a side project called InternalsAPI, which allowed me to generate interfaces to internal methods from arbitrary assemblies. Around that time I also started experimenting with AI-assisted tools.

AI helped me understand how simple reflection-based approaches could be replaced with faster delegate-based solutions. Over time I learned how to work with Unity components in a more natural way, and most of the earlier WeakReference usage disappeared.

I also spent a lot of time studying the Valheim source code. In some cases I even asked AI to analyze parts of the code to help me understand the “dialect” or coding style that the Valheim developers seem to follow. This helped me shape OTAB so that it feels closer to the original game's design philosophy.

AI was also helpful when discussing design decisions, especially when fine-tuning creature behavior and configuration data in order to keep OTAB aligned with the vanilla Valheim experience.

Typical questions I asked AI included things like:

Which version of this method is more performant, A or B?

I don't yet understand how this system works — could you show me a simple example?

Why is this done this way in the Valheim code? Wouldn't another approach be better?

How could this part of the code be optimized?

Let's talk about creature X, its Valheim lore, and possible real-world inspiration.

Is there any logical flaw in this code section?

Does this behavior remain consistent with vanilla logic?

Could you help me rewrite this text in proper English?

Learning with AI

AI was never used in the sense of “generate everything and ship it”. Instead, I treated it as a learning tool. I reviewed everything that was generated, learned from it, and often rewrote or discarded parts that did not fit the project.

In that sense, AI became more like a discussion partner, a mentor, and sometimes even a small source of motivation during moments when progress felt difficult — or simply someone to philosophize and chat with when things were not going well.

Even this text was written with the help of AI so that what I want to say is easier for others to understand.

Over time OTAB grew into a project that I genuinely fell in love with.

Inspired by the Valheim codebase, the game itself, and the many amazing mods created by other modders, I hope that OTAB can contribute something meaningful back to the community.

Design Philosophy – Staying close to Vanilla

From the beginning, OTAB was intended to stay as close as possible to the spirit of vanilla Valheim.

The developers of Valheim clearly invested a lot of care into many aspects of the game: smooth animations, satisfying sound design, logical progression systems, and a world that draws inspiration from both mythology and real-world folklore. A great deal of thoughtful work went into shaping that experience.

With OTAB, my intention was never to ignore or override that work, but rather to respect it.

During development I often spent hours discussing creature behavior, progression, and lore consistency — sometimes even with AI-assisted tools — asking questions such as: How might the original developers have approached this creature if it had been tameable? or What behavior would need to change in order for taming to make sense within the world of Valheim?

Valheim deliberately does not make every creature tameable, and that limitation felt intentional. Because of that, OTAB introduces its own set of restrictions and progression rules rather than simply enabling taming everywhere.

Many creatures can only be tamed after the biome's boss has been defeated for the first time, preserving the natural progression of the game.

Even when a creature technically can be tamed, the question often remains:

Does it actually make sense?

This is why systems such as starvation and aggression were introduced.

For example:

  • Deathsquitos can be tamed, but if they become hungry they may attack the player.
  • Fulings can be bribed and persuaded to join the player, but doing so can cost a significant amount of coins.

These systems exist to preserve the feeling that creatures remain creatures — not simply obedient pets.

I intentionally chose not to include HP loss from starvation, as that would have pushed the system into a much more punishing or hardcore direction than intended.

The goal was always to expand Valheim's systems while keeping them believable within the original game's design philosophy.

Data-driven Design

Another major influence on OTAB was the mod Wacky’s Database, which demonstrated how powerful and structured a data-driven approach can be.

Through that mod I learned how much flexibility can be achieved when gameplay data is stored in structured configuration files rather than being hardcoded directly in the mod.

Because of this, OTAB stores its default gameplay data in YAML-based data packs.

These packs can be enabled or disabled by:

  • local players
  • server administrators

The YAML files can also be modified directly, allowing servers to adjust creature behavior and mechanics without touching the mod code itself.

The default OTAB data tries to remain as vanilla-like as possible, so that it can serve as a foundation for many different play styles.

For example:

  • Some server administrators may want to create hardcore servers.
  • Others may prefer fun or chaotic server setups.
  • Many players simply want the vanilla experience with a few additional possibilities.

Another important goal was to make OTAB approachable even for players who are normally hesitant to install mods because they fear that mods might break the original game experience.

Even though OTAB is not an official expansion, I always tried to ensure that its changes still feel like Valheim — logical, progression-based, slightly mysterious, and full of small surprises.

For everything beyond that, the YAML data can be adjusted.

Multiplayer and Server Authority

In OTAB the server defines the active gameplay configuration.

This decision was also made with players in mind who are normally hesitant to install mods. Personally, I never liked having to maintain separate game folders for different modded servers.

OTAB tries to avoid that problem.

Players can keep their own local OTAB data packs — either the original ones or customized versions. When joining a server that also runs OTAB, the server's configuration is automatically loaded instead.

If the server administrator allows it, these server configurations can also be cached locally.

The player's own local data remains untouched.

When leaving the server, OTAB restores all modified creatures from internal backups so that the local game state returns to normal.

This system also allows OTAB to cooperate with other creature-modifying mods. For example, if a server uses DoOrDieMonsters, the administrator can create custom OTAB YAML configurations for those creatures.

Any creatures that already exist before OTAB loads its data are treated as "original" creatures by OTAB and are backed up and restored later.

Of course the client must still have those other mods installed.

Because of that, the server-authoritative system is only partially useful, and I am fully aware of that. Still, I chose this path, stayed on it, and simply want to see where it leads.

Data transmitted from the server to the client is encrypted. The client receives a key as well, which is also encrypted.

This system is not meant to be highly secure. Its purpose is simply to avoid accidental spoilers if a server administrator has configured something special in the OTAB data packs.

Servers running OTAB require clients to also have OTAB installed.

However, a player who has OTAB installed can still join servers that do not run OTAB.

In that case OTAB switches into a passive vanilla mode.

In this mode only UI-related features remain active, such as displaying:

  • consumable items
  • fed timers
  • pregnancy durations

These are features similar to other client-side quality-of-life mods.

Final Words

OTAB started as a small experiment.

Over time it became a project that taught me a lot — about programming, about design, and about the surprisingly complex behavior of virtual creatures.

If OTAB can add a few new stories, surprises, or interesting gameplay moments to Valheim worlds, then the project has already achieved more than I originally hoped for.
